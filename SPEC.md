# SPEC.md

## 1. 项目名称

**Codex Agent Studio**

仓库名：`codex-agent-studio`

---

## 2. 项目定位

Codex Agent Studio 是一个基于 Web 的 AI Agent 应用，用于验证“全程使用 Codex 驱动开发”的工程模式。

本项目的核心目标不是仅做一个聊天页面，而是交付一个具备以下能力的 Agent 工作台：

1. 用户可通过 Web 页面与 Agent 对话
2. Agent 可调用配置好的工具完成任务
3. 页面可展示思考过程摘要、工具调用记录、运行状态与结果
4. 支持知识导入、会话管理、结果追踪
5. 整个项目以 Codex 为主要研发执行者，用户主要负责提出需求、审核结果、验收功能

---

## 3. 项目目标

### 3.1 核心目标

在尽量不手写代码的前提下，通过 Codex 完成一个可运行、可演示、可持续迭代的 AI Agent Web 应用。

### 3.2 MVP 目标

第一阶段必须实现以下最小闭环：

- 用户打开网页后可以新建会话
- 用户可以输入问题并获得 Agent 回复
- Agent 至少支持 3 类工具能力：
  - 通用文本问答
  - 知识检索
  - 自定义函数工具调用
- 页面可以展示：
  - 对话消息
  - Agent 当前状态
  - 工具调用记录
  - 最终答案
- 系统支持基础会话持久化
- 系统具备清晰、稳定、可复用的代码结构
- 项目可本地一键启动

### 3.3 中期目标

在 MVP 跑通后，逐步扩展：

- 多工具编排
- 工具调用审批机制
- 文件上传与知识库增强
- 多 Agent 协作
- 运行 trace 与调试面板
- 用户配置管理
- 部署上线

---

## 4. 非目标（当前阶段不做）

以下内容不属于 MVP 范围：

- 多租户 SaaS 化
- 复杂 RBAC 权限系统
- 企业级计费
- 原生移动端 App
- 生产级向量平台深度集成
- 极复杂工作流编排器
- 自动写代码并直接修改生产环境
- 高风险外部系统写操作（如直接执行数据库变更、直接发邮件、直接调用真实生产 API）

## 6. 产品形态

本项目是一个 **Web 端 AI Agent 工作台**，主要由以下部分组成：

1. **聊天工作区**
   - 用于与 Agent 进行多轮对话
   - 展示回答、状态、工具调用、错误信息

2. **知识管理页**
   - 上传文本资料
   - 浏览已接入知识
   - 触发索引或检索测试

3. **运行记录页**
   - 查看每次 Agent 运行详情
   - 查看工具调用链路
   - 查看成功/失败状态

4. **设置页**
   - 配置模型
   - 配置系统提示词
   - 配置启用工具
   - 配置 Agent 行为策略

## 7. 产品原则

### 7.1 全程 Codex 驱动开发
- 用户尽量不直接写代码
- 所有实现任务通过 Codex 以自然语言下达
- 用户只做需求表达、过程审阅、结果验收、问题反馈

### 7.2 先最小可用，再逐步增强
- 优先跑通单 Agent MVP
- 不在第一阶段引入不必要的复杂性
- 不一开始做过度工程化设计

### 7.3 可观察、可解释、可追踪
- Agent 不应只是“返回一个答案”
- 必须能看到运行状态、工具调用与异常信息
- 页面应尽量让用户理解 Agent 正在做什么

### 7.4 安全优先
- 默认只允许低风险工具
- 高风险动作必须显式关闭或增加审批
- 所有工具调用都必须有日志记录

### 7.5 结构清晰，便于 Codex 维护
- 目录结构清晰
- 模块职责单一
- 命名统一
- 测试与文档齐全

---

## 8. 技术路线

### 8.1 总体路线

采用 **Web 全栈单仓库** 方案，优先降低实现复杂度。

推荐技术选型：

- **前端框架**：Next.js
- **语言**：TypeScript
- **UI**：Tailwind CSS + shadcn/ui
- **服务端 Agent 编排**：OpenAI Responses API（MVP）
- **后续可选增强**：OpenAI Agents SDK
- **数据库**：SQLite（开发环境） / PostgreSQL（后续）
- **ORM**：Prisma 或 Drizzle（二选一，由 Codex 统一实现）
- **状态管理**：优先轻量方案，避免过度复杂
- **部署**：Vercel 或 Docker 部署（二期确定）

### 8.2 为什么选择该路线

- 单仓库全栈更适合 Codex 统一生成与维护
- TypeScript 对 Web 端交互更直接
- MVP 阶段不拆前后端，减少系统复杂度
- 先以 Responses API 完成单 Agent 闭环
- 后续若进入多 Agent / MCP / tracing，再引入 Agents SDK

## 9. 核心功能需求

## 9.1 聊天会话

### 功能描述
用户可创建、切换、删除会话，并在会话中与 Agent 进行多轮对话。

### 需求明细
- 支持新建会话
- 支持会话列表展示
- 支持会话标题自动生成
- 支持会话消息持久化
- 支持多轮上下文连续对话
- 支持清空当前会话

### 验收标准
- 新建会话后可立即开始聊天
- 刷新页面后历史会话仍可访问
- 进入历史会话后能恢复上下文消息展示

---

## 9.2 Agent 消息交互

### 功能描述
用户向 Agent 提问后，系统应以接近实时的方式反馈运行状态和最终结果。

### 需求明细
- 支持用户输入问题
- 支持消息发送中状态
- 支持 Agent 回复流式展示
- 支持错误提示
- 支持重试上一轮提问
- 支持复制 Agent 回复内容

### 验收标准
- 输入后前端可立即反馈“处理中”
- 回复可逐步渲染或分阶段显示
- 失败时用户可以看到明确错误信息

---

## 9.3 工具调用能力

### 功能描述
Agent 必须具备调用工具的能力，而非仅生成文本回答。

### MVP 必须包含的工具
1. **知识检索工具**
   - 从已上传资料中检索相关内容
2. **页面内置函数工具**
   - 如总结、提取结构化信息、生成清单等
3. **安全型扩展工具**
   - 例如模拟任务规划、Mock 数据查询、演示用外部接口读取

### 二期可选工具
- Web 搜索
- 文件搜索
- 向量检索
- 第三方 API
- MCP 工具
- 工作流工具
- 审批工具

### 验收标准
- 至少有 3 类可实际调用的工具
- 用户问题触发工具时，页面能看到工具调用记录
- 工具异常时页面有失败提示和日志记录

---

## 9.4 工具调用可视化

### 功能描述
用户在 Web 界面上能看到 Agent 调用了哪些工具、调用参数摘要、执行结果摘要。

### 需求明细
- 展示工具名称
- 展示开始时间 / 结束时间
- 展示状态（运行中、成功、失败）
- 展示输入摘要
- 展示输出摘要
- 支持展开查看详情

### 验收标准
- 每次工具调用都有单独记录
- 页面可区分成功与失败调用
- 详情面板能帮助用户理解执行过程

---

## 9.5 知识库管理

### 功能描述
系统支持用户导入资料，为 Agent 提供知识上下文。

### MVP 支持
- 上传 txt / md / pdf（PDF 可先只记录并预留能力，MVP 至少确保 txt / md 可用）
- 展示知识列表
- 删除知识项
- 检索命中预览

### 二期增强
- 自动切片
- 向量检索
- 标签分类
- 文档状态管理
- 多知识空间

### 验收标准
- 用户上传文本资料后可在知识页看到
- 知识资料能被检索工具调用
- 检索结果可回显片段摘要

---

## 9.6 运行记录与追踪

### 功能描述
系统应保留 Agent 的运行记录，用于回放、排查与演示。

### 需求明细
- 保存每次提问与回答
- 保存工具调用链
- 保存错误信息
- 支持查看最近运行历史
- 支持按会话查看运行详情

### 验收标准
- 任意一次会话请求都可追溯
- 能从运行记录页查看关键执行信息
- 失败运行不会静默丢失

---

## 9.7 设置中心

### 功能描述
为后续扩展预留系统设置能力。

### MVP 支持
- 模型选择（先保留配置项）
- 系统提示词编辑
- 工具启用 / 关闭
- Agent 回复风格配置

### 验收标准
- 设置改动后对新请求生效
- 设置页面结构清晰、可扩展

## 13. Agent 行为定义

### 13.1 Agent 角色

系统内默认只有一个主 Agent：

**Studio Assistant Agent**

职责：

- 理解用户意图
- 结合上下文组织回答
- 在需要时调用工具
- 返回清晰、结构化、可信的结果
- 对自身不确定性保持诚实
- 尽量避免编造

### 13.2 Agent 决策原则

- 能直接回答时直接回答
- 需要知识依据时优先检索知识
- 需要结构化处理时调用函数工具
- 工具失败时明确告知失败原因
- 不调用未注册工具
- 不执行高风险操作
- 不伪造工具执行结果

### 13.3 回复风格原则

- 默认专业、简洁、结构化
- 用户要求详细时提供更完整说明
- 对不确定内容有兜底回复声明
- 优先给出可执行建议
- 在适合场景下输出清单、步骤、表格或 JSON

## 16. 安全与约束

### 16.1 工具安全

- 默认仅接入低风险工具
- 禁止默认执行系统命令
- 禁止默认访问生产数据库
- 禁止默认写入外部业务系统

### 16.2 输入安全

- 对用户输入进行基础校验
- 对上传内容做类型限制
- 对工具参数进行 schema 校验

### 16.3 输出安全

- 明确区分模型生成内容与工具返回内容
- 对错误堆栈进行适度脱敏
- 不在前端暴露敏感密钥

### 16.4 密钥管理

- 所有密钥走环境变量
- 不写死在代码中
- 开发环境和生产环境分离

## 19. 开发流程要求（Codex 驱动）

### 19.1 开发原则

- 所有功能尽量通过 Codex app 生成
- 人类不直接修改业务代码，除非为了紧急修复
- 每个任务必须先有明确目标，再交由 Codex 执行
- 每次提交都必须可运行、可验证

### 19.2 任务执行流程

1. 人类提出一个明确任务
2. Codex 分析需求并修改代码
3. Codex 运行测试 / lint / build
4. 人类审查 diff 与页面效果
5. 通过后提交到仓库

### 19.3 任务粒度要求

每次只让 Codex 做一个清晰目标，例如：

- 初始化项目脚手架
- 实现聊天页面
- 接入会话持久化
- 增加工具调用面板
- 实现知识上传页面

避免一次性下达过大任务。

### 19.4 每次任务的完成定义

每个任务必须满足：

- 代码已生成
- 本地可运行
- 无明显报错
- 有最基本说明
- 能被下一任务继续复用

## 21. 验收标准（总体验收）

当满足以下条件时，项目视为完成 MVP：

1. 本地可一键启动
2. 用户可在 Web 页面创建会话并与 Agent 对话
3. Agent 至少能调用 3 类工具
4. 工具调用记录可在页面上被看见
5. 会话和消息可持久化
6. 知识资料可导入并参与回答
7. 运行失败时有清晰提示
8. 代码结构清晰，便于后续继续交给 Codex 迭代
9. `README` 能指导新成员启动项目
10. 不依赖手工修补才能运行

## 22. 风险与应对

### 风险 1：Codex 一次生成代码过多，质量不稳定
应对：

- 严控任务粒度
- 先做小闭环
- 每步都验证

### 风险 2：前后端边界混乱
应对：

- 严格分层
- 先定义目录职责
- 避免把所有逻辑写进页面组件

### 风险 3：工具调用不可解释
应对：

- 工具调用必须写日志
- 前端必须有调用记录展示

### 风险 4：知识库能力过早复杂化
应对：

- MVP 先支持简单文本知识
- 向量化和复杂索引放到二期

### 风险 5：开发过程失控
应对：

- 每次任务都要有明确输入和完成定义
- 每次都让 Codex 跑 `build` / `lint` / `test`
- 所有变更通过 Git 审查
